# Go-Copilot 业务需求文档

> **版本**: v0.1  
> **状态**: 草案  
> **读者**: 开发团队、产品经理  

---

## 1. 项目概述

**Go-Copilot** 是一个面向 Go 开发者的**可定制化命令行编码 Agent 框架**，是 [`pi-ai-go`](https://github.com/pi-ai-go/pi-ai-go) 的官方示例项目。

### 1.1 一句话定位

一个跑在终端里的 Go 编程搭档——能问答、能写代码、能改文件，还能记住你的偏好和项目约定。

### 1.2 背景与动机

现有的 AI 编码工具（如 GitHub Copilot、Cursor、Claude Code）虽然强大，但存在两个痛点：

| 痛点 | 问题描述 |
|------|----------|
| **Go 语言支持不够深** | 大多数工具以 Python/TypeScript 为优先，对 Go 的类型系统、错误处理模式、工程规范理解不足 |
| **定制能力弱** | 难以注入团队内部的编码规范、项目架构约定、领域知识——工具「不知道」你的项目背景 |

Go-Copilot 旨在解决这两个问题，同时作为 `pi-ai-go` 框架的「活文档」，展示如何基于 `pi-ai-go` 构建一个功能完整的 AI Agent。

### 1.3 范围与边界

| 维度 | 明确做 | 明确不做 |
|------|--------|----------|
| **语言** | Go | 不主动支持其他语言 |
| **交互** | 终端 REPL + 单次查询 | 不提供 IDE 插件 |
| **部署** | 本地 CLI 工具 | 不做云端 SaaS 版本 |
| **定制** | 通过 Skills (Markdown) 注入领域知识 | 不提供图形化配置 |
| **集成** | 基于 `pi-ai-go` 框架 | 不替代 `pi-ai-go` 作为独立产品 |

## 2. 目标用户与场景

### 2.1 核心用户画像

| 画像 | 特征 | 核心诉求 |
|------|------|----------|
| **Go 后端工程师** | 日常使用 Go 开发微服务/中间件 | 快速问答、代码 Review、样板代码生成 |
| **Go 新手** | 正在学习 Go 语言 | 语法解释、最佳实践建议、错误排查 |
| **技术 Leader** | 负责代码质量和团队规范 | 需要助手理解团队约定，输出符合规范的代码 |

### 2.2 核心场景

| 场景 | 典型交互 | 预期产出 |
|------|----------|----------|
| **编程问答** | "这段 context 超时为什么没生效？" | 解释原因 + 修复建议 |
| **代码 Review** | "帮我看下这个 handler 有什么问题" | 指出隐患 + 优化方案 |
| **代码生成** | "生成一个带重试的 HTTP client" | 生成可编译的 `.go` 文件 |
| **代码修改** | "把这个函数改成泛型版本" | 直接修改文件并展示 diff |
| **知识注入** | 加载团队的 `SKILL.md` 后问："按我们的规范写个 middleware" | 输出符合团队规范的代码 |

### 2.3 典型工作日

```
09:00  启动 Go-Copilot，它自动加载了昨天的会话和记忆
09:05  "帮我把 logger 改成结构化日志" → 修改了 3 个文件
10:30  "这段代码并发安全吗？" → 找到了 data race
14:00  加载新同事写的 SKILL.md，"按新规范重构这个 handler" → 符合规范
17:00  "总结今天改了什么" → 列出修改摘要
```

## 3. 核心功能需求

### 3.1 Skills 技能系统（核心差异化）⚡

| 需求 | 优先级 | 说明 |
|------|--------|------|
| 从目录递归加载 `SKILL.md` | P0 | Trae 风格技能文件，注入到 system prompt |
| 支持多个技能同时激活 | P0 | 不同场景叠加不同技能 |
| 运行时热加载技能 | P1 | 无需重启即可加载/切换技能 |
| 用户自定义技能 | P0 | 开发者可自行编写 Markdown 技能文件 |
| 技能依赖与组合 | P2 | 技能间声明依赖关系 |

**用户故事**：
> 作为 Go 团队的技术 Leader，我想创建一份 `SKILL.md` 描述我们的项目架构分层规范，让 AI 助手在生成代码时自动遵循这些约定。

### 3.2 对话与持久化

| 需求 | 优先级 | 说明 |
|------|--------|------|
| 多轮对话（REPL） | P0 | 支持交互式连续对话 |
| 单次查询模式 | P0 | `-query "xxx"` 一次性执行后退出 |
| 对话持久化（JSONL） | P0 | 断电可恢复，下次启动继续对话 |
| 内存模式 | P1 | 不需要持久化时纯内存运行 |
| 对话导出 | P2 | 支持导出为 Markdown/文本 |

**用户故事**：
> 作为 Go 开发者，我下班前退出 Go-Copilot，第二天早上启动后能接上昨天的上下文继续工作。

### 3.3 长期记忆系统

| 需求 | 优先级 | 说明 |
|------|--------|------|
| 键值对记忆存储 | P0 | JSON 格式，跨会话保留 |
| 显式记忆操作 | P0 | `:remember` / `:forget` 交互命令 |
| 自动记忆触发 | P0 | 输入包含 `[remember:key=val]` 时自动保存 |
| LLM 自动提取记忆 | P2 | 每 N 轮由 LLM 判断是否有值得记住的事实 |
| 记忆缓存失效 | P0 | 记忆变更时自动刷新 system prompt |
| 记忆容量管理 | P1 | 限制总记忆大小，避免上下文溢出 |

**用户故事**：
> 我告诉 Go-Copilot 我喜欢用 `slog` 而不是 `logrus`，之后每次生成日志代码都用 `slog`。

### 3.4 上下文窗口管理

| 需求 | 优先级 | 说明 |
|------|--------|------|
| Token 估算（含中英文） | P0 | 区分 CJK 字符的 token 计算 |
| 软阈值自动压缩 | P0 | 达到 70% 上下文窗口时触发 LLM 摘要 |
| 硬阈值强制截断 | P0 | 达到 95% 时强制丢弃最早消息 |
| 手动压缩 | P0 | `:compact` 命令手动触发 |
| 上下文使用状态查看 | P0 | `:context` 命令查看 token 用量 |
| 增量 Token 统计 | P0 | O(1) 添加 vs O(N) 重算 |

**用户故事**：
> 长对话到一半时，Go-Copilot 自动把早期对话压缩成摘要，不会因为 token 超限而丢失上下文或报错。

### 3.5 LLM 集成

| 需求 | 优先级 | 说明 |
|------|--------|------|
| 多 Provider 支持 | P0 | Claude、GPT、DeepSeek、Kimi 等 |
| 可配置 API Base URL | P0 | 支持自定义代理/中转 |
| 流式输出 | P0 | 边接收边打印 |
| KeyPool 多 Key 轮询 | P1 | 多个 API Key 分摊速率限制 |
| 故障 Key 自动冷却 | P1 | Key 超限后自动暂停 60 秒 |

**用户故事**：
> 我配了 3 个 Claude API Key，Go-Copilot 自动轮换使用，不会因为单 Key 频率限制而中断工作。

### 3.6 文件操作

| 需求 | 优先级 | 说明 |
|------|--------|------|
| 代码生成 | P0 | 根据描述生成完整 `.go` 文件 |
| 代码修改 | P0 | 对现有文件进行修改并展示 diff |
| 多文件操作 | P1 | 一个请求涉及多个文件的修改 |
| 安全检查 | P0 | 文件操作前确认，支持预览 diff |
| Git 集成 | P2 | 自动 commit 修改、查看 git diff |

**用户故事**：
> 我告诉 Go-Copilot:"给所有 handler 加上 request ID middleware"，它找到所有 handler 文件并逐一修改。

### 3.7 交互系统

| 需求 | 优先级 | 说明 |
|------|--------|------|
| REPL 交互模式 | P0 | `go run .` 进入持续对话 |
| 单次查询模式 | P0 | `-query "xxx"` |
| 交互命令系统 | P0 | `:history` `:context` `:stats` 等 |
| 命令帮助 | P1 | `:help` 显示所有命令 |
| 历史回溯 | P0 | 上下方向键浏览历史输入 (*) |

## 4. 非功能需求

### 4.1 性能

| 指标 | 目标 | 当前 |
|------|------|------|
| 启动时间 | < 2 秒 | ~500ms（并行加载） |
| Token 统计开销 | O(1) 增量 | ✅ 已实现 |
| LLM 首 token 延迟 | < 3 秒 | 取决于 API |
| 响应流式输出间隔 | < 100ms/块 | ✅ |
| Session 写入延迟 | 异步不阻塞 | ✅ |

### 4.2 可用性

| 指标 | 要求 |
|------|------|
| 学习成本 | 5 分钟内完成首次查询 |
| 安装步骤 | 3 步以内（clone → 配 key → run） |
| 错误恢复 | 网络中断后支持重试，Key 故障自动切换 |
| 内存占用 | < 100MB（不含模型） |

### 4.3 安全

| 要求 | 说明 |
|------|------|
| API Key 不落日志 | 调试输出中自动脱敏 |
| 记忆本地存储 | 不主动上传用户数据 |
| 文件操作需确认 | 修改文件前给用户预览 |

## 8. 风险与假设

### 8.1 关键假设

| 假设 | 风险 | 缓解 |
|------|------|------|
| 用户有自己的 LLM API Key | 门槛高，部分用户无 Key | 未来考虑内置免费额度 |
| 用户了解命令行工具 | Go 新手可能不熟悉 | 提供清晰的 README 和 `:help` |
| Go 语言生态持续增长 | 用户基数有限 | 保持专注比泛化更重要 |
| `pi-ai-go` 框架稳定 | 上游变更影响本项目 | 固定依赖版本，CI 自动检测 |

### 8.2 主要风险

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| LLM API 不稳定/涨价 | 中 | 高 | 多 Provider 支持 + KeyPool |
| Token 消耗成本高 | 高 | 中 | 上下文压缩 + 增量统计优化 |
| 竞争产品快速迭代 | 高 | 中 | 聚焦 Go + Skills 差异化 |
| 开源社区参与不足 | 中 | 中 | 作为 `pi-ai-go` 官方示例，天然有曝光 |
