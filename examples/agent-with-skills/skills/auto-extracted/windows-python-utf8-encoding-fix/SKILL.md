# windows-python-utf8-encoding-fix

> **Trigger:** Python script outputs Emoji (📅👥🔗) or Unicode characters not in the system code page,
> and crashes with `UnicodeEncodeError: 'gbk' codec can't encode character` on Windows terminals.

## Steps

1. **Identify the crash location** — determine which `print()` calls or logging output produce the characters that cause the encode error. Look for Emoji, special symbols, or non-ASCII text in format strings.

2. **Add stdout/stderr reconfiguration** — at the top of the module, before any output occurs, insert:
   ```python
   import sys

   if sys.stdout.encoding != "utf-8":
       sys.stdout.reconfigure(encoding="utf-8", errors="replace")
   if sys.stderr.encoding != "utf-8":
       sys.stderr.reconfigure(encoding="utf-8", errors="replace")
   ```
   The `encoding` check prevents unnecessary reconfiguration on already-UTF-8 terminals. `errors="replace"` ensures unrepresentable characters render as `?` instead of crashing.

3. **Verify cross-terminal compatibility** — test the script in both Windows `cmd` and PowerShell environments where the system code page is GBK (936) or another non-UTF-8 encoding. Confirm output is printed without exceptions, even if some characters appear as boxes.

4. **Maintain the fix as an isolated block** — keep the encoding fix in a clearly commented block near the top of the entry-point script. Do not mix it with business logic or scatter it across files. Use a visible separator comment like `# ── Encoding fix: Windows GBK → UTF-8 ──`.

## Output

- A Python script that can output Emoji and Unicode characters on Windows terminals without crashing.
- Clear, commented encoding-reconfiguration block at the top of the affected module.
- No change to output on POSIX systems where `sys.stdout.encoding` is already `utf-8`.
